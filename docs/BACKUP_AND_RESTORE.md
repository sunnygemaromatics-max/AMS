# Backup & Restore

The application has three independent state surfaces that must each be
backed up separately:

1. **Source code** — GitHub repository
2. **Database** — Supabase PostgreSQL (tables + auth.users)
3. **Hostinger files** — the deployed `dist/` content under `public_html/`

A complete restore on a fresh stack requires #1 + #2. #3 is regenerated
from #1 by the build.

---

## 1. Source code backup

### Already in place

- The GitHub repository itself is the canonical backup.
- Tagged restore points (see `git tag -l`) — notably:
  - `v1.0.0` — production release.
  - `handover-backup-2026-05-16-pre-cleanup` — full state captured
    immediately before the production cleanup / history rewrite.

### How to take an additional snapshot

```bash
# 1. Tag the current commit
git tag -a "snapshot-$(date +%F)" -m "Snapshot before <reason>"
git push origin "snapshot-$(date +%F)"

# 2. Offline archive (everything except node_modules / dist)
git archive --format=zip --prefix=ams/ HEAD -o ams-$(date +%F).zip
```

### Restore

```bash
git fetch --tags
git reset --hard <tag>
git push --force origin main      # triggers a redeploy via Actions
```

## 2. Supabase database backup

### Automated daily backups

Supabase Pro and above include **daily automated backups** of the entire
database, retained for 7 days (Pro) / 14 days (Team) / 30 days (Enterprise).
Restore from **Supabase Dashboard → Database → Backups → "Restore"** —
this provisions the snapshot into a new project; you then re-point the app.

For the **Free tier**, manual backups are required.

### Manual database export (works on any tier)

**Option A — Supabase CLI** (recommended for scheduled / automated backups):

```bash
npm install -g supabase
supabase login
supabase link --project-ref <your-project-ref>

# Full dump (schema + data + auth)
supabase db dump --file backup-$(date +%F).sql

# Data only
supabase db dump --data-only --file backup-data-$(date +%F).sql
```

**Option B — pg_dump direct** (if you have the database password):

```bash
PGPASSWORD='<db-password>' pg_dump \
  -h db.<project-ref>.supabase.co -p 5432 -U postgres -d postgres \
  -F c -f backup-$(date +%F).dump
```

Get the host / port / DB name from Supabase Dashboard → **Project Settings
→ Database → Connection string**.

### Restoring a manual dump

To a **new Supabase project**:

```bash
# SQL dump:
psql <new-project-connection-string> < backup.sql

# Custom format:
pg_restore -d <new-project-connection-string> backup.dump
```

After restore:
1. Update `VITE_SUPABASE_URL` and `VITE_SUPABASE_PUBLISHABLE_KEY` in
   GitHub Actions Secrets to the new project's URL/key.
2. Push any commit to `main` to redeploy the frontend pointing at the new DB.

### What to back up regularly

- **Critical:** the entire `public` schema (your business data) and
  `auth.users` / `auth.identities` (login accounts).
- **Recoverable from code:** Edge Functions (they live in `supabase/functions/`),
  RLS policies (recreated by the migration scripts), Storage bucket
  configuration (see below).

### Recommended cadence

| Tier | Recommendation |
|---|---|
| Free | Weekly manual `supabase db dump`, kept off-server. |
| Pro+ | Rely on automated daily snapshots; take a manual dump before any major migration or destructive operation. |

## 3. Hostinger files backup

```
Hostinger File Manager → public_html → Select all → Compress → save as
backup-public_html-YYYY-MM-DD.zip in the same folder (or download it).
```

This is your fastest rollback path — restore takes ~30 seconds (delete
current files, upload + extract the zip). Take this snapshot **before each
deploy** when not relying on CI.

The deployed files can also always be regenerated by running the build
against the corresponding git tag.

---

## 4. Supabase storage buckets

If you add image uploads / file attachments later (the current build has no
storage buckets in active use), back them up via:

```bash
supabase storage download --recursive <bucket-name> ./bucket-backup-$(date +%F)
```

Bucket **policies** (RLS for storage) are recreated from SQL — keep the
defining SQL in `supabase/migrations/` so they are versioned with the
codebase.

## 5. Environment variables backup

`.env` is **not** in git (correctly — it contains secrets). Maintain it in a
team password manager (1Password, Bitwarden, LastPass). The variables you
must preserve:

| Variable | Where it is needed |
|---|---|
| `VITE_SUPABASE_URL` | local `.env` + GitHub Actions Secrets |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | local `.env` + GitHub Actions Secrets |
| `VITE_SENTRY_DSN` (if used) | local `.env` + GitHub Actions Secrets |
| `HOSTINGER_FTP_SERVER` / `_USERNAME` / `_PASSWORD` | GitHub Actions Secrets only |
| `SUPABASE_SERVICE_ROLE_KEY` (if Edge Functions used) | Supabase Function secrets only |
| `RESEND_API_KEY` / `ALERT_FROM_EMAIL` (if email alerts used) | Supabase Function secrets only |

### Service role key — special handling

The Supabase `service_role` key bypasses RLS. **Never** commit it, never
ship it in the browser bundle. It belongs in:

- `supabase secrets set` for Edge Functions
- Server-side cron / pg_cron for scheduled jobs
- Nowhere else.

If a service role key is ever exposed: **rotate it immediately** in
Supabase Dashboard → Project Settings → API → "Reset service role key".

## 6. Disaster recovery — full rebuild from zero

If everything is lost (repo + project + files), recovery from the
documented backups:

1. **Create a new GitHub repository** and push from your local clone or the
   `handover-backup-*` archive zip.
2. **Create a new Supabase project**, run migrations:
   ```bash
   supabase link --project-ref <new-ref>
   supabase db push          # applies supabase/migrations/*
   ```
   Then run, in numeric order, every file in `database/*.sql` via the
   SQL Editor (see `database/README.md`).
3. **Restore the most recent database dump** into the new project (Section 2).
4. **Re-add all GitHub Actions Secrets** (Section 5 table).
5. **Push any commit** — CI builds and deploys to Hostinger automatically.
6. Verify via the health-check list in `docs/DEPLOYMENT.md`.

Estimated total recovery time from a complete loss: **~45 minutes**.
